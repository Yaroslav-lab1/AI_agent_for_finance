# QA по проекту AI Agent for Finance

## 1. Что делает проект?

Проект помогает пользователю вести личные финансы: вручную добавлять операции, импортировать банковские уведомления из текста, CSV и картинок, получать предварительные кандидаты транзакций, проверять их и подтверждать в основную таблицу операций.

## 2. Какая главная идея AI-агента?

AI-агент не сразу пишет данные в финальные транзакции. Он сначала извлекает кандидатов, нормализует сумму и дату, определяет тип операции, подбирает категорию, проверяет дубли и отдаёт пользователю preview. Это снижает риск ошибочной записи.

## 3. Какие источники импорта поддерживаются?

- `text`: текст банковских уведомлений.
- `csv`: табличный экспорт банка.
- `image`: скриншот или фото уведомления.

## 4. Как работает импорт картинки?

Картинка проходит цепочку:

1. Backend принимает файл на `/api/imports/image`.
2. `ImageOcrTool` проверяет MIME-тип и размер.
3. Pillow открывает картинку, переводит её в grayscale, усиливает контраст и при необходимости увеличивает размер.
4. Tesseract OCR извлекает текст.
5. Полученный OCR-текст нормализуется как обычный текст.
6. Текст передаётся в LLM-парсер.
7. LLM возвращает structured output по JSON Schema.
8. Пайплайн создаёт кандидатов импорта.

## 5. Почему не сохраняем OCR-текст сразу как транзакции?

OCR может ошибиться в символах, переносах строк, валюте или названии магазина. Поэтому после OCR используется LLM-парсер, который восстанавливает смысл уведомления и возвращает структурированные поля.

## 6. Какой LLM используется?

Проект поддерживает несколько провайдеров:

- `mock`: быстрый тестовый парсер.
- `ollama`: локальная модель, сейчас `qwen3.5:9b`.
- `openai`: OpenAI-compatible API при наличии ключа.

Текущая локальная настройка использует Ollama и модель `qwen3.5:9b`.

## 7. Что такое structured output?

Это режим, при котором модель должна вернуть JSON строго заданной структуры: массив `transactions`, а внутри сумма, дата, тип операции, категория, комментарий и confidence. Это уменьшает вероятность свободного текста вместо машинно обрабатываемого результата.

## 8. Где находится JSON Schema для ответа модели?

В `backend/app/agent/llm_client.py`, константа `TRANSACTION_EXTRACTION_SCHEMA`.

## 9. Почему используется PostgreSQL?

PostgreSQL хранит пользователей, категории, транзакции, задания импорта, кандидатов импорта и audit log. Для проекта важны связи между таблицами, транзакционность и нормальные SQL-запросы для статистики.

## 10. Для чего нужен PostgreSQL пароль?

Пароль нужен для аутентификации пользователя базы данных. Его использует:

- контейнер PostgreSQL при создании пользователя;
- backend в `DATABASE_URL`, чтобы подключаться к БД;
- человек, если подключается к БД через `psql`, DBeaver, TablePlus или pgAdmin.

Пароль не является паролем пользователя приложения. Это пароль технического пользователя БД.

## 11. Где посмотреть актуальное содержимое БД?

Вариант через терминал:

```powershell
docker compose exec postgres psql -U finance_agent -d finance_agent
```

Полезные команды внутри `psql`:

```sql
\dt
select * from users;
select * from transactions;
select * from import_jobs;
select * from import_candidates;
select * from agent_audit_logs;
```

Вариант через GUI-клиент:

- host: `localhost`
- port: `5432`
- database: значение `POSTGRES_DB` из `.env`
- user: значение `POSTGRES_USER` из `.env`
- password: значение `POSTGRES_PASSWORD` из `.env`

## 12. Где физически лежат данные PostgreSQL?

В Docker volume `postgres_data`, который объявлен в `docker-compose.yml`. Это значит, что данные переживают перезапуск контейнеров, пока volume не удалён.

## 13. Что будет, если удалить volume?

База очистится. При следующем старте backend заново применит Alembic-миграции и создаст схему, но пользовательские данные исчезнут.

## 14. Зачем нужны Alembic-миграции?

Они версионируют схему базы. Вместо ручного создания таблиц проект может воспроизводимо применить изменения структуры БД в Docker, локально и на другой машине.

## 15. Какие основные таблицы есть?

- `users`: пользователи.
- `categories`: категории доходов и расходов.
- `transactions`: подтверждённые операции.
- `import_jobs`: задания импорта.
- `import_candidates`: кандидаты, найденные агентом.
- `agent_audit_logs`: журнал работы импортов.

## 16. Чем кандидат импорта отличается от транзакции?

Кандидат - это предложенная агентом операция, которую пользователь ещё может исправить или отклонить. Транзакция - подтверждённая операция, попавшая в финальный финансовый учёт.

## 17. Как проверяются дубли?

Пайплайн строит `source_hash` по сумме, типу операции, дате и комментарию. Потом ищет совпадения среди транзакций пользователя и помечает кандидата как точный или возможный дубль.

## 18. Почему категории дополнительно классифицируются после LLM?

LLM может вернуть неточную или нестандартную категорию. Поэтому есть `CategoryClassifierTool`, который приводит результат к известным slug-значениям из базы.

## 19. Как устроена авторизация?

Пользователь регистрируется по email/password. При логине backend выдаёт JWT. Frontend отправляет JWT в `Authorization: Bearer ...`, а backend через dependency получает текущего пользователя.

## 20. Почему используется preview перед подтверждением?

Финансовые данные чувствительны к ошибкам. Preview даёт пользователю контроль: проверить сумму, дату, категорию и комментарий перед созданием настоящей транзакции.

## 21. Как проверить backend вручную?

Healthcheck:


Swagger/OpenAPI:

```text

```

## 22. Как запустить тесты?

Локально:

```powershell
cd backend
python -m pytest
```

В Docker:

```powershell
docker compose exec -T backend pytest
```

## 23. Почему тесты принудительно используют mock LLM?

Чтобы тесты были быстрыми, стабильными и не зависели от локально запущенной Ollama-модели. Реальную модель проверяем отдельным smoke-test.

## 24. Как доказать, что картинки идут через OCR в LLM?

Есть тест `test_image_payload_is_ocr_text_before_llm` в `backend/tests/unit/test_agent_tools.py`. Он мокает OCR, перехватывает вход LLM и проверяет, что LLM получил именно нормализованный OCR-текст.

## 25. Какие ограничения есть у текущего OCR?

- Плохое качество скриншота может дать ошибки распознавания.
- Tesseract не понимает смысл уведомления, он только извлекает символы.
- Для прямого vision-парсинга без OCR нужна мультимодальная модель, например Qwen-VL, а текущий `qwen3.5:9b` используется как text LLM.

## 26. Что можно улучшить дальше?

- Сохранять OCR-текст в audit log для отладки.
- Добавить предобработку изображений: deskew, crop, adaptive threshold.
- Добавить отдельный confidence для OCR.
- Поддержать vision-модель для картинок низкого качества.
- Добавить end-to-end тест с реальным маленьким PNG.
