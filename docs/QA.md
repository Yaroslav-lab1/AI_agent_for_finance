# QA по проекту

Проект: **Веб-приложение для учета личных финансов с AI-агентом**.

Документ помогает быстро вспомнить устройство проекта, подготовиться к защите и ответить на типовые вопросы по архитектуре, функциональности, стеку и ограничениям.

## 1. Что делает приложение?

Приложение помогает пользователю вести учет личных финансов: регистрироваться, входить в личный кабинет, добавлять доходы и расходы вручную, импортировать операции из текста, CSV и изображений, смотреть список операций, фильтровать, сортировать и анализировать расходы по категориям.

## 2. В чем главная ценность проекта?

Проект уменьшает ручной ввод финансовых операций. Пользователь может вставить банковское уведомление, загрузить CSV или скриншот, а AI-агент подготовит кандидатов операций. При этом финальное решение остается за пользователем.

## 3. Какая главная идея AI-импорта?

AI-импорт не записывает найденные операции сразу в финальную таблицу. Сначала он создает кандидатов: пользователь видит предварительный результат, может исправить сумму, тип, категорию, дату и комментарий, а затем подтверждает только нужные операции. Это снижает риск ошибочной записи финансовых данных.

## 4. Какие источники импорта поддерживаются?

- `text`: текст банковских уведомлений или выписки.
- `csv`: табличный файл с банковскими операциями.
- `image`: PNG, JPEG или WebP со скриншотом уведомлений.

## 5. Как обрабатывается изображение?

Изображение проходит через OCR: backend принимает файл, `ImageOcrTool` проверяет MIME-тип и размер, Pillow открывает и предварительно обрабатывает картинку, Tesseract извлекает текст, а затем этот текст идет в тот же AI-пайплайн, что и обычный текстовый импорт.

## 6. Почему картинка сначала идет через OCR, а не сразу в LLM?

Текущая модель `qwen3.5:9b` используется как текстовая LLM. Она хорошо подходит для смыслового разбора текста, но не является vision-моделью. Поэтому изображение сначала превращается в текст через OCR, а уже потом LLM извлекает из него структурированные операции.

## 7. Как работает CSV-импорт?

`CsvInputTool` читает CSV-файл, определяет разделитель, нормализует строки и приводит данные к единому текстовому виду. После этого результат передается в общий AI-пайплайн. За счет этого текст, CSV и OCR-текст обрабатываются единообразно.

## 8. Какие CSV-колонки понимает импорт?

Инструмент ищет типовые названия колонок: дата, сумма, описание/комментарий, тип операции и категория. Поддерживаются русские и английские алиасы, например `date`, `amount`, `comment`, `merchant`, `дата`, `сумма`, `описание`, `категория`.

## 9. Какой LLM-провайдер используется?

Проект поддерживает три режима:

- `mock`: тестовый парсер без реальной LLM.
- `ollama`: локальная модель через Ollama, например `qwen3.5:9b`.
- `openai`: OpenAI-compatible API, который можно использовать в том числе через OpenRouter при настройке `OPENAI_BASE_URL` и ключа.

## 10. Что такое structured output в проекте?

Это требование к LLM вернуть JSON по заданной схеме. В ответе должны быть операции с суммой, датой, типом, категорией, комментарием и confidence. Схема находится в `backend/app/agent/llm_client.py` в константе `TRANSACTION_EXTRACTION_SCHEMA`.

## 11. Почему structured output важен?

Без structured output модель могла бы вернуть произвольный текст, который сложно надежно обработать. JSON Schema ограничивает форму ответа и упрощает backend-валидацию перед созданием кандидатов.

## 12. Что делает `AgentPipeline`?

`AgentPipeline` объединяет все этапы AI-импорта: подготовку входа, вызов LLM, нормализацию суммы и даты, определение типа операции, подбор категории, проверку дублей и формирование кандидатов для предпросмотра.

## 13. Какие инструменты входят в AI-пайплайн?

- `TextInputTool`: нормализует текст.
- `CsvInputTool`: превращает CSV в читаемый текстовый вход.
- `ImageOcrTool`: извлекает текст из изображения.
- `OllamaLLMClient` или `OpenAICompatibleLLMClient`: вызывает LLM.
- `AmountNormalizerTool`: приводит сумму к числу.
- `DateNormalizerTool`: приводит дату к формату приложения.
- `OperationTypeResolverTool`: уточняет доход или расход.
- `CategoryClassifierTool`: подбирает категорию.
- `DuplicateDetectorTool`: помечает возможные дубли.

## 14. Почему кандидаты импорта хранятся отдельно от операций?

Кандидат - это предположение AI-агента. Его можно исправить, отклонить или подтвердить. Операция в `transactions` - это уже финальная запись пользователя. Такое разделение делает импорт безопаснее и прозрачнее.

## 15. Как проверяются дубли?

Пайплайн строит `source_hash` по ключевым полям операции: сумме, типу, дате и комментарию. Затем backend ищет совпадения среди операций пользователя и помечает кандидата как точный или возможный дубль.

## 16. Сохраняется ли история работы агента?

Да. Для каждого импорта создается `import_job`, а ход выполнения фиксируется в `agent_audit_logs`: источник, статус, размер входа, количество кандидатов, длительность и ошибка при неудаче.

## 17. Какие основные таблицы есть в PostgreSQL?

- `users`: аккаунты пользователей.
- `categories`: категории доходов и расходов.
- `transactions`: подтвержденные операции.
- `import_jobs`: задания AI-импорта.
- `import_candidates`: найденные кандидаты.
- `agent_audit_logs`: журнал работы AI-пайплайна.

## 18. Для чего нужен пароль PostgreSQL?

Это пароль технического пользователя базы данных. Его используют контейнер PostgreSQL и backend при подключении через `DATABASE_URL`. Он не связан с паролем пользователя приложения.

## 19. Где посмотреть содержимое базы данных?

Через терминал:

```powershell
docker compose exec postgres psql -U finance_agent -d finance_agent
```

Полезные команды внутри `psql`:

```sql
\dt
select * from users;
select * from categories;
select * from transactions;
select * from import_jobs;
select * from import_candidates;
select * from agent_audit_logs;
```

Также можно подключиться через DBeaver, TablePlus или pgAdmin:

- host: `localhost`
- port: `5432`
- database: значение `POSTGRES_DB`
- user: значение `POSTGRES_USER`
- password: значение `POSTGRES_PASSWORD`

## 20. Как устроена авторизация?

Пользователь регистрируется по имени, логину, email и паролю. При входе backend выдает JWT. Frontend хранит токен и отправляет его в заголовке `Authorization: Bearer ...`. Backend через dependency получает текущего пользователя и изолирует данные по `user_id`.

## 21. Почему пароль пользователя хранится безопасно?

Backend не хранит исходный пароль. Пароль хешируется через `bcrypt`, а при входе сравнивается новый ввод пользователя с сохраненным хешем.

## 22. Как работает редактирование аккаунта?

Пользователь может изменить имя, логин, email и пароль в панели профиля. Для любого изменения нужно указать текущий пароль от аккаунта. Backend проверяет пароль, валидирует уникальность email и логина, а новый пароль сохраняет только в виде хеша.

## 23. Какие операции можно делать вручную?

Пользователь может создать, изменить и удалить операцию. Для операции задаются сумма, тип, категория, дата и комментарий. Ручные операции сохраняются с источником `manual`.

## 24. Как устроен список операций?

Список операций поддерживает фильтры по датам, типу, категории и поисковой строке. Сортировка выполняется кликами по заголовкам таблицы: дата, тип, сумма, категория, комментарий и источник. По умолчанию операции отсортированы по дате по убыванию.

## 25. Как устроена статистика?

Backend считает сводку по доходам, расходам и балансу. На frontend есть блок "Расходы по категориям" с двумя кольцевыми диаграммами: за все время и за выбранный месяц. В центре каждой диаграммы показывается баланс за соответствующий период.

## 26. Почему категории в легенде диаграмм отсортированы по сумме?

Так пользователь сначала видит самые значимые категории расходов. Цвета категорий единые для обеих диаграмм, поэтому легенда показывается один раз под графиками.

## 28. Как запустить проект?

Через Docker Compose из корня проекта:

```powershell
docker compose up --build
```

После запуска:

- frontend: `http://localhost:5173`
- backend API: `http://localhost:8000/api`
- Swagger: `http://localhost:8000/docs`

## 29. Как включить локальный qwen через Ollama?

В `.env` нужно указать:

```env
LLM_PROVIDER=ollama
LLM_MODEL=qwen3.5:9b
OLLAMA_BASE_URL=http://host.docker.internal:11434
```

Для локального запуска backend без Docker URL может быть `http://localhost:11434`.

## 30. Как подключить OpenRouter?

OpenRouter подключается через OpenAI-compatible режим:

```env
LLM_PROVIDER=openai
OPENAI_BASE_URL=https://openrouter.ai/api/v1
OPENAI_API_KEY=...
LLM_MODEL=...
```

Код пайплайна при этом менять не нужно.

## 31. Как запустить backend-тесты?

Локально:

```powershell
cd backend
python -m pytest
```

В Docker:

```powershell
docker compose exec -T backend pytest
```

## 32. Почему тесты используют mock LLM?

Тесты должны быть быстрыми и стабильными. Реальная локальная LLM зависит от запущенной Ollama, модели и мощности компьютера. Поэтому основная проверка логики использует mock, а реальный LLM-парсинг проверяется отдельно вручную или benchmark-запуском.

## 33. Какие проверки покрыты тестами?

Проверяются healthcheck, регистрация, вход, обновление профиля с текущим паролем, конфликты email и логина, CRUD операций, user scope, статистика, текстовый импорт, CSV-импорт, image-импорт с mock OCR, подтверждение кандидатов, дубли и отдельные инструменты агента.

## 34. Насколько быстрый AI-импорт?

Базовый benchmark с реальным `qwen3.5:9b` показал:

| Режим | 1 операция | 5 операций | 10 операций |
|---|---:|---:|---:|
| Текст | 2.88 с | 9.97 с | 19.00 с |
| CSV | 4.26 с | 10.79 с | 19.92 с |
| Изображение | 2.76 с | 10.39 с | 20.33 с |

Основное время занимает LLM-инференс. OCR, CSV-парсинг и backend-логика добавляют меньшую часть задержки.

## 35. Какие ограничения есть у текущего OCR?

OCR может ошибаться на размытых, темных, повернутых или низкоконтрастных изображениях. Tesseract извлекает символы, но не понимает бизнес-смысл уведомления. Поэтому после OCR используется LLM-парсер и пользовательский предпросмотр.

## 36. Есть ли сейчас нормализация транслита названий магазинов?

Нет, отдельный слой нормализации транслита в текущей версии не используется. Если в банковском уведомлении магазин записан латиницей, качество категоризации зависит от LLM и простых правил `CategoryClassifierTool`. Возможное развитие - добавить словарь merchant aliases и fuzzy matching.

## 37. Какие ограничения есть у текущего AI-подхода?

- Модель может ошибаться в категории или комментарии.
- OCR может испортить входной текст.
- Локальная LLM требует установленной модели и ресурсов компьютера.
- Синхронный импорт больших пачек может заставить пользователя ждать десятки секунд.

## 38. Какие планы развития проекта?

- Улучшить внешний вид и варианты диаграмм.
- Добавить асинхронную очередь импортов для больших файлов.
- Добавить словарь магазинов и нормализацию транслита.
- Перейти с локального Ollama на LLM через OpenRouter для более портативного запуска.
- Добавить LLM в роли финансового консультанта.
- Расширить личный кабинет настройками пользователя и предпочтениями отображения.
