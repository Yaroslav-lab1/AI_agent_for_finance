# Легенда 1: Backend API и доменная логика

## Роль

Участник отвечал за backend-слой проекта: REST API, бизнес-логику операций, авторизацию, профиль пользователя, статистику и связь с базой данных.

## Что сделал

- Спроектировал основные доменные сущности backend: `User`, `Category`, `Transaction`, `ImportJob`, `ImportCandidate`, `AgentAuditLog`.
- Реализовал FastAPI-роуты для регистрации, входа, получения текущего пользователя и обновления профиля.
- Добавил правило безопасности: для изменения имени, логина, email или пароля пользователь должен указать актуальный пароль.
- Реализовал проверку уникальности email и username при регистрации и обновлении профиля.
- Реализовал CRUD операций: создание вручную, список с фильтрами, обновление и удаление.
- Реализовал user scope: пользователь видит и меняет только свои операции, импорты и кандидатов.
- Реализовал API статистики: общий доход, расход, баланс и расходы по категориям с фильтром по периоду.
- Реализовал backend-часть подтверждения AI-кандидатов: выбранные кандидаты превращаются в записи `transactions`, остальные могут быть отклонены.
- Разделил роуты, сервисы, репозитории, схемы и модели, чтобы HTTP-слой не смешивался с бизнес-логикой.

## Что может рассказать на защите

Главная идея backend: финальные финансовые записи и AI-кандидаты разделены. AI может ошибаться, поэтому backend сначала сохраняет кандидатов импорта, а пользователь подтверждает только нужные операции. Такой подход безопаснее для финансового приложения.

Также можно объяснить, что все основные пользовательские данные привязаны к `user_id`. Это защищает от ситуации, когда один пользователь случайно получает операции или импорты другого.

## Ответственность за качество

Проверял регистрацию, вход, обновление профиля с текущим паролем, конфликты email/логина, изоляцию данных по пользователю, CRUD операций, фильтры, статистику, обработку ошибок API и корректное создание финальных операций из подтвержденных кандидатов.

## Возможный вопрос на защите

**Почему нельзя сразу записывать результат AI в `transactions`?**

Потому что AI и OCR могут ошибаться. Кандидаты дают пользователю контроль: он видит результат, исправляет поля и подтверждает только корректные операции.
