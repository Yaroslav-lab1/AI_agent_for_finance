# Обоснование библиотек и технологий

Короткий формат: зачем инструмент в проекте, два плюса, два минуса и почему он подходит для задачи.

## FastAPI

Зачем: backend API для авторизации, транзакций, статистики и импортов.

Плюсы:
- Высокая скорость разработки API за счёт Python type hints и Pydantic-схем.
- Автоматическая OpenAPI-документация удобна для тестирования и защиты.

Минусы:
- Async-код требует аккуратной работы с сессиями БД и зависимостями.
- Для production всё равно нужны настройки observability, rate limits и секретов.

Доказательство актуальности: официальная документация описывает FastAPI как современный high-performance framework для API на Python type hints: https://fastapi.tiangolo.com/.

## PostgreSQL

Зачем: основная реляционная БД для пользователей, категорий, транзакций, импортов и audit logs.

Плюсы:
- Надёжная транзакционная БД с отношениями, индексами и SQL-запросами.
- Хорошо подходит для финансовых данных, где важна целостность.

Минусы:
- Требует администрирования: пароли, volume, миграции, backup.
- Для простого прототипа тяжелее, чем SQLite.

Доказательство актуальности: PostgreSQL позиционируется как мощная open-source object-relational database system с многолетним развитием и репутацией надёжности: https://www.postgresql.org/about/.

## SQLAlchemy Async ORM

Зачем: работа backend-кода с БД через модели и async-сессии.

Плюсы:
- Позволяет описывать таблицы как Python-модели.
- Поддерживает asyncio для FastAPI-приложения.

Минусы:
- ORM может скрывать сложность SQL-запросов.
- Async SQLAlchemy имеет больше нюансов, чем синхронный режим.

Доказательство актуальности: официальная документация SQLAlchemy описывает поддержку asyncio для Core и ORM: https://docs.sqlalchemy.org/en/latest/orm/extensions/asyncio.html.

## Alembic

Зачем: миграции схемы БД.

Плюсы:
- Версионирует изменения таблиц.
- Хорошо интегрируется с SQLAlchemy.

Минусы:
- Миграции надо поддерживать дисциплинированно.
- Autogenerate не заменяет ручную проверку миграций.

Доказательство актуальности: Alembic официально описан как lightweight database migration tool для SQLAlchemy: https://alembic.sqlalchemy.org/.

## Pydantic и pydantic-settings

Зачем: схемы запросов/ответов API и конфигурация из `.env`.

Плюсы:
- Runtime validation на основе type hints.
- Может генерировать JSON Schema, что полезно для structured output.

Минусы:
- Валидация добавляет небольшой overhead.
- Нужно следить за различиями версий Pydantic v1/v2.

Доказательство актуальности: документация Pydantic подчёркивает validation/serialization через type annotations, скорость ядра на Rust и JSON Schema: https://pydantic.dev/docs/validation/latest/get-started/.

## Ollama

Зачем: локальный запуск LLM `qwen3.5:9b` без облачного API.

Плюсы:
- Данные остаются локально, что важно для финансовых уведомлений.
- Structured outputs позволяют требовать ответ по JSON Schema.

Минусы:
- Локальные модели могут отвечать медленно на слабом железе.
- Качество зависит от модели и промпта.

Доказательство актуальности: Ollama documented structured outputs allow enforcing a JSON schema for reliable extraction: https://docs.ollama.com/capabilities/structured-outputs.

## qwen3.5:9b

Зачем: локальный LLM-парсер банковских уведомлений.

Плюсы:
- Достаточно компактная модель для локального запуска.
- Подходит для смыслового разбора текста после OCR.

Минусы:
- Не является vision-моделью, поэтому картинки идут через OCR.
- Может ошибаться в редких форматах банковских сообщений.

Доказательство эффективности в проекте: smoke-test через `/api/imports/text` вернул 3 кандидата из 3 банковских списаний; Ollama structured output удерживает формат ответа по JSON Schema.

## Tesseract OCR и pytesseract

Зачем: извлечение текста из изображений перед LLM-парсингом.

Плюсы:
- Open-source OCR engine с поддержкой множества языков.
- Хорошо подходит для печатного текста на скриншотах уведомлений.

Минусы:
- Ошибается на размытых, наклонённых и низкоконтрастных фото.
- Не понимает бизнес-смысл текста, только распознаёт символы.

Доказательство актуальности: официальная документация Tesseract описывает его как open source OCR engine для извлечения printed text from images и поддержку широкого набора языков: https://tesseract-ocr.github.io/tessdoc/.

## Pillow

Зачем: открытие и предобработка изображений перед OCR.

Плюсы:
- Простая работа с форматами PNG/JPEG/WebP.
- Можно быстро улучшить изображение: grayscale, contrast, resize.

Минусы:
- Не решает сложные задачи computer vision.
- Нужна ручная настройка предобработки под тип изображений.

Доказательство эффективности в проекте: предобработка повышает читаемость OCR-входа без добавления тяжёлых CV-зависимостей.

## React

Зачем: frontend-интерфейс для авторизации, транзакций, импорта и статистики.

Плюсы:
- Компонентная модель удобна для страниц, форм и списков.
- Большая экосистема библиотек для состояния, графиков и UI.

Минусы:
- Требует сборки и tooling.
- Без дисциплины компоненты могут стать перегруженными.

Доказательство актуальности: официальная документация React описывает построение UI из компонентов: https://react.dev/.

## Vite

Зачем: dev server и сборка frontend.

Плюсы:
- Быстрый старт dev server.
- Удобная современная сборка React-приложения.

Минусы:
- Нужна отдельная Node.js-инфраструктура.
- Production-конфигурацию всё равно надо проверять отдельно.

Доказательство актуальности: документация Vite описывает его как fast frontend build tool и выделяет instant server start/HMR: https://vite.dev/.

## Axios

Зачем: HTTP-клиент frontend для обращения к backend API.

Плюсы:
- Удобная работа с JSON, headers и ошибками.
- Promise-based API подходит для React Query и async UI.

Минусы:
- Дублирует часть возможностей встроенного `fetch`.
- Добавляет зависимость в frontend bundle.

Доказательство актуальности: Axios описывает себя как promise based HTTP client for browser and node.js: https://axios-js.com/docs/index.html.

## TanStack React Query

Зачем: загрузка, кэширование и обновление серверных данных на frontend.

Плюсы:
- Уменьшает ручной код для loading/error/refetch.
- Удобен для API-first приложений.

Минусы:
- Нужно понимать stale time, invalidation и query keys.
- Для маленького проекта может показаться лишним слоем.

Доказательство эффективности в проекте: интерфейс часто читает и обновляет серверные сущности: транзакции, категории, статистику и импорты.

## Recharts

Зачем: визуализация расходов по категориям.

Плюсы:
- Быстро строит графики в React.
- Достаточно прост для учебного dashboard.

Минусы:
- Менее гибок, чем низкоуровневые D3-подходы.
- Большие кастомизации могут усложнить код.

Доказательство эффективности в проекте: статистика расходов лучше воспринимается визуально, а Recharts закрывает эту задачу без отдельной BI-системы.

## Docker Compose

Зачем: единый запуск backend, frontend и PostgreSQL.

Плюсы:
- Весь стек описан в одном YAML.
- Удобно поднимать проект одной командой.

Минусы:
- Нужно понимать порты, volumes и переменные окружения.
- Локальная среда может отличаться от production.

Доказательство актуальности: Docker docs описывают Compose как инструмент для defining and running multi-container applications: https://docs.docker.com/compose/.

## pytest

Зачем: unit и API-тесты backend.

Плюсы:
- Лаконичные readable tests.
- Масштабируется от маленьких unit-тестов до функциональных тестов.

Минусы:
- Async fixtures требуют аккуратной настройки.
- Моки могут скрыть проблемы реальных интеграций, поэтому нужны smoke-tests.

Доказательство актуальности: официальная документация pytest говорит, что framework makes it easy to write small, readable tests and can scale to complex functional testing: https://docs.pytest.org/.

## httpx

Зачем: тестовый async HTTP-клиент для API-тестов FastAPI.

Плюсы:
- Позволяет тестировать ASGI-приложение без реального сетевого сервера.
- Поддерживает async-клиент.

Минусы:
- Тестовая среда отличается от настоящего браузера.
- Нужно отдельно проверять CORS и frontend-интеграцию.

Доказательство эффективности в проекте: API-тесты проверяют реальные роуты `/api/...` через ASGITransport и выполняются быстро.
