# Обоснование технологий

Документ фиксирует актуальный стек проекта и объясняет, почему выбранные инструменты подходят для веб-приложения учета личных финансов с AI-импортом банковских операций.

## Backend: Python 3.12 + FastAPI + Uvicorn

Зачем используется: backend предоставляет REST API для авторизации, профиля пользователя, операций, статистики, категорий и AI-импорта.

Плюсы:

- FastAPI хорошо работает с type hints, dependency injection и Pydantic-схемами.
- Автоматическая OpenAPI-документация удобна для демонстрации и ручной проверки API.

Минусы:

- Async-код требует аккуратной работы с сессиями БД и внешними HTTP-вызовами.
- Для production нужны дополнительные настройки логирования, rate limit, мониторинга и секретов.

Почему подходит: проект API-first, frontend отделен от backend, а FastAPI быстро закрывает сценарий "typed REST API + OpenAPI + async I/O". Uvicorn используется как ASGI-сервер для запуска FastAPI-приложения.

## Валидация и настройки: Pydantic + pydantic-settings

Зачем используется: Pydantic описывает схемы запросов и ответов, а `pydantic-settings` читает настройки из `.env`.

Плюсы:

- Runtime validation снижает риск записать некорректные финансовые данные.
- JSON Schema применяется и в API, и в structured output для LLM.

Минусы:

- Валидация добавляет небольшой runtime overhead.
- Нужно учитывать особенности сериализации `Decimal`, `UUID`, дат и nullable-полей.

Почему подходит: в проекте много структурированных сущностей: пользователь, операция, категория, import job, кандидат импорта, статистика и LLM-ответ.

## Безопасность: JWT, bcrypt, python-jose, email-validator

Зачем используется: реализует регистрацию, вход, хранение паролей и проверку доступа к данным пользователя.

Плюсы:

- `bcrypt` хранит не исходный пароль, а устойчивый хеш.
- JWT через `python-jose` позволяет frontend передавать токен в `Authorization: Bearer ...`.

Минусы:

- JWT нельзя "отозвать" без дополнительного blacklist/rotation-механизма.
- Безопасность зависит от качества `JWT_SECRET`, срока жизни токена и HTTPS в production.

Почему подходит: приложению нужна простая stateless-авторизация. Любое изменение профиля требует текущий пароль, что снижает риск случайной или чужой смены email, логина, имени и пароля.

## ORM и доступ к БД: SQLAlchemy Async + asyncpg

Зачем используется: backend работает с PostgreSQL через async ORM-модели, репозитории и сервисы.

Плюсы:

- Таблицы описываются как Python-модели, что упрощает поддержку доменной схемы.
- Async-доступ хорошо сочетается с FastAPI и внешними LLM/OCR-вызовами.

Минусы:

- ORM может скрывать сложность SQL-запросов.
- Async SQLAlchemy требует дисциплины: сессии, транзакции и lazy-loading надо контролировать.

Почему подходит: проект хранит связанные данные пользователей, операций, импортов и кандидатов, поэтому ORM удобнее ручного SQL в каждом обработчике.

## Миграции БД: Alembic

Зачем используется: версионирует изменения схемы PostgreSQL.

Плюсы:

- Позволяет воспроизводимо создать структуру БД на другой машине.
- Хорошо интегрируется с SQLAlchemy-моделями.

Минусы:

- Миграции нужно поддерживать аккуратно и проверять вручную.
- Autogenerate не заменяет понимание изменений схемы.

Почему подходит: в проекте несколько связанных таблиц, а схема развивается вместе с функциональностью.

## Database: PostgreSQL 16

Зачем используется: хранит пользователей, категории, подтвержденные операции, задания импорта, кандидатов и audit log.

Плюсы:

- Надежная транзакционная реляционная БД с внешними ключами и индексами.
- Хорошо подходит для финансовых данных, где важны целостность и предсказуемые запросы.

Минусы:

- Нужны администрирование, пароль, volume и миграции.
- Для простого прототипа тяжелее, чем SQLite.

Почему подходит: проекту нужны связи между пользователями, категориями и операциями, а также безопасное хранение истории импортов и кандидатов до подтверждения.

## Загрузка файлов: python-multipart

Зачем используется: FastAPI принимает CSV и изображения через multipart/form-data.

Плюсы:

- Стандартный способ загрузки файлов из браузера.
- Хорошо совместим с FastAPI `UploadFile`.

Минусы:

- Требует ограничивать размер файлов.
- Нужно отдельно проверять MIME-тип и содержимое файла.

Почему подходит: AI-импорт принимает не только текст, но и CSV/изображения, поэтому multipart-загрузка естественно ложится на текущий API.

## Frontend: React 18 + TypeScript + Vite

Зачем используется: реализует интерфейс авторизации, профиля, dashboard, ручного ввода, AI-импорта, таблицы операций и графиков.

Плюсы:

- Компонентная модель удобна для форм, таблиц, панелей, кандидатов импорта и повторяемых UI-блоков.
- TypeScript помогает согласовать frontend-типы с API.

Минусы:

- Нужна сборка и Node.js tooling.
- При росте страницы компоненты нужно дробить, иначе dashboard становится сложным.

Почему подходит: приложение интерактивное, с множеством состояний: авторизация, профиль, фильтры, сортировки, формы, кандидаты импорта, диаграммы и загрузка файлов.

## HTTP и серверное состояние: Axios + TanStack Query

Зачем используется: frontend обращается к backend API, кэширует данные и обновляет UI после мутаций.

Плюсы:

- Axios удобен для JSON, multipart-загрузок и заголовка `Authorization`.
- TanStack Query упрощает loading/error/refetch и invalidation после создания, изменения или удаления данных.

Минусы:

- Появляется дополнительный слой абстракции поверх `fetch`.
- Нужно аккуратно проектировать query keys и сброс кэша.

Почему подходит: dashboard часто читает и обновляет серверные сущности: операции, категории, статистику, профиль и импорты.

## Визуализация: Recharts

Зачем используется: строит кольцевые диаграммы расходов по категориям.

Плюсы:

- Быстро интегрируется с React.
- Достаточно гибок для donut chart, tooltip, единой legend и центрального текста баланса.

Минусы:

- Для сложной аналитики менее гибок, чем прямой D3-подход.
- Кастомизация нестандартных графиков может усложнить компонент.

Почему подходит: проекту нужны понятные dashboard-графики без отдельной BI-системы. Текущая версия показывает две диаграммы: за все время и за выбранный месяц.

## AI: Ollama, qwen3.5:9b и OpenAI-compatible client

Зачем используется: LLM извлекает операции из банковского текста и возвращает structured output по JSON Schema.

Плюсы:

- Ollama позволяет запускать локальную модель и не отправлять финансовые тексты во внешний сервис.
- OpenAI-compatible режим позволяет перейти на OpenRouter или другой совместимый API без переписывания пайплайна.

Минусы:

- Локальная модель зависит от ресурсов компьютера и может работать медленно.
- Качество результата зависит от модели, промпта, OCR и формата исходных данных.

Почему подходит: банковские уведомления бывают разными по формату, поэтому LLM полезна для смыслового разбора текста. Structured output снижает риск свободного неструктурированного ответа. `qwen3.5:9b` выбран как локальный компромисс между качеством, мультиязычностью и возможностью запуска на учебном стенде.

## OCR: Tesseract + pytesseract + Pillow

Зачем используется: извлекает текст из скриншотов банковских уведомлений перед передачей в AI-парсер.

Плюсы:

- Tesseract - зрелый open-source OCR-движок с поддержкой нескольких языков.
- Pillow позволяет выполнить базовую предобработку изображения: открыть файл, привести к grayscale, усилить контраст, изменить размер.

Минусы:

- OCR ошибается на размытых, повернутых и низкоконтрастных изображениях.
- Tesseract извлекает символы, но не понимает финансовый смысл текста.

Почему подходит: текущая LLM является текстовой, поэтому OCR - практичный промежуточный слой для импорта изображений. В Docker backend устанавливает системные пакеты `tesseract-ocr` и `tesseract-ocr-rus`.

## Инфраструктура: Docker Compose

Зачем используется: поднимает PostgreSQL, backend и frontend одной командой.

Плюсы:

- Описывает весь локальный стенд в одном `docker-compose.yml`.
- Упрощает запуск проекта на другой машине.

Минусы:

- Нужно понимать ports, volumes и переменные окружения.
- Production-развертывание потребует отдельной настройки.

Почему подходит: для учебного fullstack-проекта важно быстро воспроизводить окружение с БД, API, интерфейсом, Ollama-настройками и OCR-зависимостями.

## Тестирование: pytest, pytest-asyncio, httpx, aiosqlite

Зачем используется: проверяет backend-сервисы, API и инструменты агента.

Плюсы:

- `pytest` дает простой формат тестов и фикстур.
- `httpx` и `pytest-asyncio` позволяют тестировать async FastAPI без запуска отдельного сервера.

Минусы:

- Тестовая БД на SQLite не полностью повторяет PostgreSQL.
- Реальная LLM в тестах заменяется mock-клиентом, поэтому качество модели проверяется отдельно.

Почему подходит: тесты покрывают основную backend-логику быстро и стабильно, не завися от Ollama.

## Производительность AI-импорта

Базовое нагрузочное тестирование проводилось на реальном `qwen3.5:9b` через Ollama. Для каждого размера было по два прогона, warm-up не учитывался.

| Режим | 1 операция | 5 операций | 10 операций |
|---|---:|---:|---:|
| Текст | 2.88 с | 9.97 с | 19.00 с |
| CSV | 4.26 с | 10.79 с | 19.92 с |
| Изображение | 2.76 с | 10.39 с | 20.33 с |

Вывод: задержка почти линейно растет от числа операций. Основное время занимает LLM-инференс; CSV-парсинг, backend-логика и OCR добавляют существенно меньшую долю времени. Для больших пачек операций логичным следующим шагом будет асинхронная очередь импортов.
