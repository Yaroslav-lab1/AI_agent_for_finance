# Обоснование технологий

Документ фиксирует актуальный стек проекта и объясняет, почему выбранные инструменты подходят для веб-приложения учета личных финансов с AI-импортом.

## Backend: Python + FastAPI

Зачем используется: backend предоставляет REST API для авторизации, профиля, операций, статистики, категорий и AI-импорта.

Плюсы:

- Быстрое создание API за счет Python type hints, dependency injection и Pydantic-схем.
- Автоматическая OpenAPI-документация удобна для ручной проверки и демонстрации.

Минусы:

- Async-код требует аккуратной работы с сессиями БД и внешними HTTP-вызовами.
- Для production нужны дополнительные настройки безопасности, логирования и rate limit.

Почему подходит: проект API-first, а FastAPI хорошо закрывает сценарий "тонкий frontend + typed backend + OpenAPI".

## Валидация API: Pydantic и pydantic-settings

Зачем используется: описывает схемы запросов и ответов, валидирует входные данные, читает настройки из `.env`.

Плюсы:

- Runtime validation снижает риск записать некорректные финансовые данные.
- JSON Schema удобно использовать и в API, и в structured output для LLM.

Минусы:

- Валидация добавляет небольшой runtime overhead.
- Нужно учитывать различия версий и особенности сериализации дат, UUID и Decimal.

Почему подходит: в проекте много структурированных сущностей: пользователь, операция, кандидат импорта, статистика и LLM-ответ.

## ORM и доступ к БД: SQLAlchemy Async + asyncpg

Зачем используется: backend работает с PostgreSQL через async ORM-модели, репозитории и сервисы.

Плюсы:

- Таблицы описываются как Python-модели, что упрощает поддержку доменной схемы.
- Async-доступ хорошо сочетается с FastAPI.

Минусы:

- ORM может скрывать сложность SQL-запросов.
- Async SQLAlchemy требует дисциплины: сессии, транзакции и lazy-loading надо контролировать.

Почему подходит: проект хранит связанные данные пользователей, операций, импортов и кандидатов, поэтому ORM удобнее ручного SQL в каждом обработчике.

## Миграции БД: Alembic

Зачем используется: версионирует изменения схемы PostgreSQL.

Плюсы:

- Позволяет воспроизводимо создать структуру БД на другой машине.
- Хорошо интегрируется с SQLAlchemy-моделями.

Минусы:

- Миграции нужно поддерживать аккуратно и проверять вручную.
- Autogenerate не заменяет понимание изменений схемы.

Почему подходит: в проекте уже есть несколько связанных таблиц, и схема развивается вместе с функциональностью.

## Database: PostgreSQL 16

Зачем используется: хранит пользователей, категории, операции, задания импорта, кандидатов и audit log.

Плюсы:

- Надежная транзакционная реляционная БД с внешними ключами и индексами.
- Хорошо подходит для финансовых данных, где важны целостность и предсказуемые запросы.

Минусы:

- Нужны администрирование, пароль, volume и миграции.
- Для простого прототипа тяжелее, чем SQLite.

Почему подходит: проекту нужны связи между пользователями, категориями и операциями, а также безопасное хранение истории импортов.

## Frontend: React + TypeScript + Vite

Зачем используется: реализует интерфейс авторизации, профиля, dashboard, ручного ввода, AI-импорта, таблицы операций и графиков.

Плюсы:

- Компонентная модель удобна для форм, таблиц, панелей и повторяемых UI-блоков.
- TypeScript помогает согласовать frontend-типы с API.

Минусы:

- Нужна сборка и Node.js tooling.
- При росте страницы компоненты нужно дробить, иначе dashboard становится сложным.

Почему подходит: приложение интерактивное, с множеством состояний: фильтры, сортировки, формы, кандидаты импорта и графики.

## HTTP и серверное состояние: Axios + TanStack Query

Зачем используется: frontend обращается к backend API, кэширует данные и обновляет UI после мутаций.

Плюсы:

- Axios удобен для JSON, multipart-загрузок и заголовка `Authorization`.
- TanStack Query упрощает loading/error/refetch и invalidation после создания или удаления данных.

Минусы:

- Появляется дополнительный слой абстракции поверх `fetch`.
- Нужно аккуратно проектировать query keys и сброс кэша.

Почему подходит: dashboard часто читает и обновляет серверные сущности: операции, категории, статистику, профиль и импорты.

## Визуализация: Recharts

Зачем используется: строит кольцевые диаграммы расходов по категориям.

Плюсы:

- Быстро интегрируется с React.
- Достаточно гибок для donut chart, legend, tooltip и центрального текста баланса.

Минусы:

- Для сложной аналитики менее гибок, чем прямой D3-подход.
- Кастомизация нестандартных графиков может усложнить компонент.

Почему подходит: проекту нужны понятные dashboard-графики без отдельной BI-системы.

## AI: Ollama и OpenAI-compatible client

Зачем используется: LLM извлекает операции из банковского текста и возвращает structured output по JSON Schema.

Плюсы:

- Ollama позволяет запускать локальную модель и не отправлять финансовые тексты во внешний сервис.
- OpenAI-compatible режим позволяет перейти на OpenRouter или другой совместимый API без переписывания пайплайна.

Минусы:

- Локальная модель зависит от ресурсов компьютера и может работать медленно.
- Качество результата зависит от модели, промпта и формата исходных данных.

Почему подходит: банковские уведомления бывают разными по формату, поэтому LLM полезна для смыслового разбора текста, а structured output снижает риск свободного неструктурированного ответа.

## OCR: Tesseract + pytesseract + Pillow

Зачем используется: извлекает текст из скриншотов банковских уведомлений перед передачей в AI-парсер.

Плюсы:

- Tesseract - зрелый open-source OCR-движок с поддержкой нескольких языков.
- Pillow позволяет быстро выполнить базовую предобработку изображения: открыть файл, привести к grayscale, усилить контраст, изменить размер.

Минусы:

- OCR ошибается на размытых, повернутых и низкоконтрастных изображениях.
- Tesseract извлекает символы, но не понимает финансовый смысл текста.

Почему подходит: текущая LLM является текстовой, поэтому OCR - практичный промежуточный слой для импорта изображений.

## Инфраструктура: Docker Compose

Зачем используется: поднимает PostgreSQL, backend и frontend одной командой.

Плюсы:

- Описывает весь локальный стенд в одном `docker-compose.yml`.
- Упрощает запуск проекта на другой машине.

Минусы:

- Нужно понимать ports, volumes и переменные окружения.
- Production-развертывание потребует отдельной настройки.

Почему подходит: для учебного fullstack-проекта важно быстро воспроизводить окружение с БД, API и интерфейсом.

## Тестирование: pytest, pytest-asyncio, httpx, aiosqlite

Зачем используется: проверяет backend-сервисы, API и инструменты агента.

Плюсы:

- `pytest` дает простой формат тестов и фикстур.
- `httpx` и `pytest-asyncio` позволяют тестировать async FastAPI без запуска отдельного сервера.

Минусы:

- Тестовая БД на SQLite не полностью повторяет PostgreSQL.
- Реальная LLM в тестах заменяется mock-клиентом, поэтому качество модели проверяется отдельно.

Почему подходит: тесты покрывают основную backend-логику быстро и стабильно, не завися от Ollama.
